description: Detect project stack and save .skynex/project-config.yaml — run once per project

Set up this project. Detects the full stack and conventions, asks three quick questions about workflow preferences, and saves `.skynex/project-config.yaml` and `.skynex/review-rules.md` so every agent session starts with project context ready.

## Step 1 — Detect project context

Read these files **if they exist** (never fail if missing):
- `CONVENTIONS.md`, `AGENTS.md` — project conventions and context
- `package.json` — runtime, framework, scripts (test/build/lint/typecheck)
- `go.mod`, `go.sum` — Go module and dependencies
- `pyproject.toml`, `requirements.txt` — Python stack
- `Cargo.toml` — Rust stack
- `tsconfig.json` — TypeScript compiler settings and path aliases
- `src/`, `cmd/`, `app/`, `internal/` top-level — architecture pattern and module layout
- Test files in `src/` or `tests/` — test framework and conventions
- `.eslintrc*`, `eslint.config.*`, `.prettierrc*`, `golangci.yml` — linter config
- `.env.example` — environment variables
- `Makefile` — build/test/lint targets if no package.json scripts

From this, infer:
- `runtime`: node | go | python | rust | ruby | ...
- `language`: typescript | javascript | go | python | rust | ...
- `framework`: nestjs | express | fastapi | gin | react | vue | svelte | none | ...
- `test_runner`: vitest | jest | playwright | go-test | pytest | cargo-test | none
- `architecture`: ddd | mvc | modular | monolith | ...
- `commands.test`: exact command string (e.g. `pnpm test`, `go test ./...`, `pytest`)
- `commands.build`: exact command or omit if none found
- `commands.lint`: exact command or omit if none found
- `commands.typecheck`: exact command or omit if not applicable

## Step 2 — Ask three questions

Ask **all three questions in a single message**, never one at a time:

> **1. Modo de trabajo**
> ¿Cómo querés que trabajen los agentes?
> - `interactivo` — pausa entre fases para que revises y apruebes (recomendado)
> - `automático` — corre todo de corrido sin pausas
>
> **2. Presupuesto de review**
> ¿Cuántas líneas cambiadas máximo antes de sugerir dividir en PRs encadenadas?
> Si `.skynex/project-config.yaml` ya existe, mostrá el valor actual como referencia. Default si no hay config: `400`
>
> **3. Reglas de review iniciales (opcional)**
> ¿Hay reglas de proyecto que querés que los agentes verifiquen en cada PR?
> Dejá en blanco para usar el template basado en el stack detectado. Editables siempre en `.skynex/review-rules.md`.
> Ejemplos: "no console.log en producción" · "todo endpoint nuevo lleva test de authz" · "antes de crear UI revisá /src/design-system/"

Wait for the user's answers before continuing.

## Step 3 — Write `.skynex/project-config.yaml`

Create `.skynex/` if it doesn't exist.
Write (or overwrite) `.skynex/project-config.yaml` — omit fields that don't apply:

```yaml
# Generated by /setup — re-run to regenerate
version: 1
updated_at: "YYYY-MM-DD"

stack:
  runtime: <detected>
  language: <detected>
  framework: <detected>
  test_runner: <detected>
  architecture: <detected>

commands:
  test: "<detected>"
  build: "<detected>"
  lint: "<detected>"
  typecheck: "<detected>"

  mode: <user answer>           # interactive | automatic
  review_budget: <user answer>  # number (default 400)

paths:
  conventions: CONVENTIONS.md   # only if file exists
  specs: docs/specs
```

## Step 4 — Write `.skynex/review-rules.md`

These rules are enforced by `/review-pr` — a violation is always a finding.

- **File already exists** → do NOT overwrite. The user's rules are sacred. Just note it in the confirmation.
- **File does not exist** → generate it with a stack-appropriate starter template.

Template to generate (write this as a clean `.md` file — adapt the "Code quality" section to only include bullets matching the detected stack, replace `{review_budget}` with the user's answer from question 2):

```
# Review Rules
# Enforced by /review-pr as findings. Edit freely.
# /review-pr suggests new rules when it detects repeated patterns — paste them here to lock them in.

## General
- No secrets, credentials, or API keys committed to source.
{if user provided rules in question 3: add each one as a bullet here}

## Code quality
{include only the bullets that match the detected language/framework:}
[TypeScript/JS]  - No `console.log` in production code — use the project logger.
[TypeScript]     - No `any` type without a comment justifying why.
[NestJS]         - All non-public endpoints must have an auth guard.
[NestJS]         - Validate input at the DTO boundary, not inside services.
[Go]             - Wrap errors with context: fmt.Errorf("doing X: %w", err).
[Go]             - No panic in library or service code.
[Python]         - No bare `except:` — catch specific exception types.
[React/Vue]      - Check existing components directory before creating a new one.
[any]            - Input validation must happen at the system boundary, not deep in business logic.

## Tests
- Behavior changes must ship at least one test covering the new path.
- Test names describe intent: "should <behavior> when <condition>".
- No mocks that prevent tests from exercising any real logic.

## PRs
- PR description must state what changed and why — not just a ticket number.
- Keep PRs under {review_budget} lines changed; split larger changes into chained PRs.
```

## Step 5 — Confirm

Show a brief summary (≤12 lines total):
- Stack and architecture detected
- Commands found
- Workflow settings saved (mode + review_budget)
- `project-config.yaml` written (or overwritten — mention if overwritten)
- `review-rules.md` created with stack-tailored starter template OR already exists (preserved — not touched)

Tell the user: re-run `/setup` anytime to regenerate `project-config.yaml`. The `review-rules.md` will never be overwritten by `/setup` — it belongs to the user.

## Rules
- Ask ALL THREE questions together in one message — never separately
- Only write values you actually detected — never invent or guess
- Omit yaml fields that are not applicable to the stack
- If `.skynex/project-config.yaml` already exists, overwrite it (mention this to the user); on rerun, read and display the existing `review_budget` value in question 2 for reference
- NEVER overwrite `.skynex/review-rules.md` if it already exists
- In the generated review-rules template, include ONLY the code quality bullets that match the detected stack (do not dump all bullets)
- Keep the confirmation summary concise — no more than 12 lines
